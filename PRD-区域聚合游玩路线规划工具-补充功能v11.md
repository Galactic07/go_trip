## 1. 文档概述

本文档是 v1.0 基础版 PRD 的**补充扩展**，聚焦「让工具从"能用"变成"好用"」的增量功能。所有功能按 **实施优先级** 分为三层：

| 层级 | 定位 | 核心特征 |
|------|------|---------|
| **T0 立即可做** | 低成本高回报，不改核心架构 | 状态管理、时间预算、链接解析、天气感知 |
| **T1 适度进阶** | 需要少量后端/算法投入 | 顺路推荐、弹性路线、轻协作 |
| **T2 长期愿景** | 产品形态升级，可作为后续大版本方向 | 城市记忆库、周期发现、外部工具联动 |

## 2. T0 层：立即可做（强烈推荐 MVP + 1 阶段落地）

### 2.1 F-11 地点状态管理

| 属性 | 描述 |
|------|------|
| **功能描述** | 为每个心愿地点增加状态标签，区分"还没去 / 已去过 / 待定 / 已关闭"等状态 |
| **用户价值** | 心愿单随时间膨胀后会变得混乱，状态管理让"整理"变成"归归档"，而非"删掉重来" |
| **优先级** | T0（必做） |
| **详细规则** | |

#### 2.1.1 状态定义

| 状态 | 标识 | 说明 | 默认值 |
|------|------|------|--------|
| 想去 | 🔵 `wishlist` | 默认状态，刚添加的地点 | ✅ |
| 已去过 | ✅ `visited` | 用户标记已打卡 | — |
| 待定 | ⏳ `pending` | 兴趣不大或信息不足，留着再说 | — |
| 已关闭/不想去 | ❌ `abandoned` | 店关了 / 不感兴趣了 | — |

#### 2.1.2 交互设计

- **标记在地图上的表现**：

| 状态 | 样式处理 |
|------|---------|
| 想去 | 正常显示，蓝色标记 |
| 已去过 | 半透明 60% + 绿色边框 + 右下角 ✓ 角标 |
| 待定 | 半透明 40% + 灰色 |
| 已关闭 | 半透明 30% + 红色删除线覆盖 |

- **侧边面板**：
  - 顶部增加状态筛选器：「全部 / 想去 / 已去过 / 待定」
  - 选择状态后列表实时过滤
  - 路线规划时默认 **排除「已去过」和「已关闭」** 的地点

- **批量操作**：
  - 支持「全部标记为已去过」「清空已关闭地点」等快捷操作
  - 批量操作前弹出确认提示

#### 2.1.3 数据模型变更

```sql
-- places 表新增字段
ALTER TABLE `places` ADD COLUMN `status` VARCHAR(20) NOT NULL DEFAULT 'wishlist' COMMENT '地点状态: wishlist/visited/pending/abandoned';
ALTER TABLE `places` ADD COLUMN `visited_at` DATETIME DEFAULT NULL COMMENT '实际到访时间';
ALTER TABLE `places` ADD INDEX `idx_status` (`status`);
```

```typescript
// TypeScript 类型扩展
type PlaceStatus = 'wishlist' | 'visited' | 'pending' | 'abandoned';

interface Place {
  // ... 原有字段
  status: PlaceStatus;
  visited_at: string | null;  // ISO 8601 时间字符串
}
```

### 2.2 F-12 路线时间预算与费用估算

| 属性 | 描述 |
|------|------|
| **功能描述** | 在路线规划中增加每个地点的预计停留时长、总耗时和粗略费用估算 |
| **用户价值** | 用户最关心的不是"先去哪后去哪"，而是"一天能不能走得完" |
| **优先级** | T0（必做） |
| **详细规则** | |

#### 2.2.1 停留时长设置

- 每个地点支持设置 **预计停留时长**，预设选项：
  - 🚶 路过打卡（15分钟）
  - ☕ 短暂停留（30分钟）
  - 📸 正常游玩（1小时）
  - 🍽️ 吃饭/深度体验（2小时）
  - 🏛️ 半天以上（3小时+）
- 用户可自定义输入任意时长（分钟级精度）
- 默认停留时长：1小时（正常游玩）

#### 2.2.2 路线概览卡片

路线生成后顶部展示汇总信息：

```
┌───────────────────────────────────────────────┐
│  🗺️ 粤海街道一日游                           │
│  ─────────────────────────────────────────    │
│  📍 6个地点  ·  🕐 游玩约5h  ·  🚶 交通约1h   │
│  💰 约 ¥180（门票+餐饮）                      │
│  ⏰ 预计总耗时：约 6-7 小时（含交通+游玩）      │
│                                               │
│  ① 欢乐海岸  ── 🚶15min ── ② 深圳湾公园 ...   │
│                                               │
│  [ 🔄 重新排序 ]  [ 💾 保存路线 ]              │
└───────────────────────────────────────────────┘
```

#### 2.2.3 费用估算

- 每个地点可填写 **人均费用**（整数，单位元），预设分类：
  - 免费（0元）
  - 低消费（≤50元）
  - 中等消费（50-200元）
  - 高消费（>200元）
- 路线汇总时自动求和，显示区间（如"约 ¥100-300"）
- 费用字段非必填，不填则不计入汇总

#### 2.2.4 数据模型变更

```sql
-- places 表新增字段
ALTER TABLE `places` ADD COLUMN `stay_duration` INT UNSIGNED NOT NULL DEFAULT 60 COMMENT '预计停留时长（分钟），默认60';
ALTER TABLE `places` ADD COLUMN `estimated_cost` INT UNSIGNED DEFAULT NULL COMMENT '预估人均费用（元）';
```

### 2.3 F-13 分享链接自动解析

| 属性 | 描述 |
|------|------|
| **功能描述** | 用户粘贴小红书/抖音/大众点评等平台的分享链接，系统自动解析出地点名称和地址，一键添加 |
| **用户价值** | 将"复制链接 → 打开APP → 记住名字 → 回来搜索"的四步流程缩短为"粘贴 → 确认"一步 |
| **优先级** | T0（强烈推荐） |
| **详细规则** | |

#### 2.3.1 支持的链接类型

| 平台 | 链接特征 | 解析方式 |
|------|---------|---------|
| 小红书 | `https://www.xiaohongshu.com/explore/...` | 后端爬虫解析页面标题+地址 |
| 抖音 | `https://www.douyin.com/video/...` | 后端解析视频描述中的POI信息 |
| 大众点评 | `https://www.dianping.com/shop/...` | 后端解析店铺名称和地址 |
| 高德地图 | `https://uri.amap.com/search?keyword=...` | 直接提取keyword参数 |
| 通用URL | 非上述平台 | 提取URL中的文本作为搜索关键词 |

#### 2.3.2 交互流程

```
1. 用户在搜索框粘贴链接（自动检测URL格式）
2. 搜索框右侧出现「🔗 解析链接」按钮，同时显示加载动画
3. 解析成功后，搜索框自动填入解析到的地点名称
4. 联想列表置顶显示"✅ 已识别：华侨城创意文化园 - 深圳市南山区"
5. 用户点击确认 → 正常添加流程
6. 若解析失败 → 提示"无法识别链接中的地点，请手动输入名称"
```

#### 2.3.3 技术实现

- **后端新增接口**：`POST /api/v1/parse-url`
  - 请求体：`{ "url": "https://..." }`
  - 响应体：`{ "title": "华侨城创意文化园", "address": "深圳市南山区锦绣北街2号", "success": true }`
- **实现策略**：
  - 使用 Go 的 `net/http` 发起请求，解析 HTML 的 Open Graph 标签（`og:title`、`og:description`）
  - 对已知平台（小红书、大众点评）做专项解析规则
  - 设置 3 秒超时，超时后返回解析失败
  - 使用缓存（内存/Redis），相同 URL 不重复请求

#### 2.3.4 边界情况

| 场景 | 处理方案 |
|------|---------|
| 链接已失效/404 | 提示"链接无法访问，请手动输入地点名称" |
| 解析到的名称不准确 | 用户可在搜索框中编辑修正后再确认 |
| 非地点类链接（如文章） | 提示"该链接未包含地点信息" |
| 网络超时 | 自动重试一次，仍失败则提示用户手动输入 |

### 2.4 F-14 天气感知与室内外标签

| 属性 | 描述 |
|------|------|
| **功能描述** | 为每个地点标记室内/室外属性，并在路线规划中结合天气预报给出出行建议 |
| **用户价值** | 避免"兴冲冲出发，到了才发现是户外，下雨了"的尴尬 |
| **优先级** | T0（低成本高惊喜） |
| **详细规则** | |

#### 2.4.1 室内外标签

- 每个地点可设置 **场景类型**（单选）：
  - 🌤 **户外为主**：公园、景点、街区、海滩等
  - 🏠 **室内为主**：商场、博物馆、咖啡馆、室内游乐场等
  - 🌦 **半户外**：有遮蔽的街区、集市、檐下空间等
- 标签来源：
  - **自动推断**：根据高德POI类型做初步判断（如"公园"→户外，"商场"→室内）
  - **用户可修正**：在地点详情编辑中手动调整
- 标签展示：在标记弹窗和侧边面板地点名称旁显示图标

#### 2.4.2 天气感知提示

- **实现方式**：
  - 使用免费天气 API（如 Open-Meteo，无需 API Key，完全免费）
  - 根据路线中地点的城市，获取当日天气预报
  - 前端定时刷新（每小时一次），或用户手动点击刷新
- **提示逻辑**：

| 天气条件 | 路线中户外比例 | 提示内容 |
|---------|--------------|---------|
| 晴天 ☀️ | ≥50% 户外 | ✅ 今天天气适合户外游玩！ |
| 雨天 🌧 | ≥50% 户外 | ⚠️ 今日有雨，路线中 4/6 为户外地点，建议调整或携带雨具 |
| 高温 ≥35°C | ≥50% 户外 | 🔥 高温预警，户外地点较多，建议早晚出行或增加室内替代 |
| 低温 ≤5°C | ≥50% 户外 | 🥶 天气较冷，户外地点建议穿着保暖 |

- **提示展示位置**：路线规划视图顶部，天气图标 + 一行文案
- **不强制阻断**：仅为建议，用户可忽略

#### 2.4.3 数据模型变更

```sql
-- places 表新增字段
ALTER TABLE `places` ADD COLUMN `scene_type` VARCHAR(20) NOT NULL DEFAULT 'unknown' COMMENT '场景类型: outdoor/indoor/hybrid/unknown';
```

### 3.1 F-15 顺路推荐

| 属性 | 描述 |
|------|------|
| **功能描述** | 在用户已规划的路线中，基于当前路线节点自动推荐"顺路可去"的周边 POI |
| **用户价值** | 从"只去想去的地方"升级为"发现附近还有什么值得顺路看看的" |
| **优先级** | T1 |
| **详细规则** | |

#### 3.1.1 推荐触发

- 用户在路线规划模式下，点击某个路线节点
- 系统以该节点为中心，搜索 **500m 半径**内的高德 POI
- 搜索结果过滤掉已经在心愿单中的地点

#### 3.1.2 推荐展示

```
┌───────────────────────────────────────────────┐
│  📍 深圳湾公园                               │

│  + 渔人码头咖啡       🚗 驾车8min    🏠 室内   │
│                                               │
│  [ + 添加到路线 ]  [ 🔄 换一批 ]              │
└───────────────────────────────────────────────┘
```

#### 3.1.3 推荐类型标签

| 标签 | 含义 | 视觉标识 |
|------|------|---------|
| 🔁 顺路 | 距离当前节点 ≤500m，不绕路 | 绿色 |

| ⭐ 高分 | 大众点评/小红书评分 ≥4.5 | 金色 |

#### 3.1.4 技术实现

- 前端标注推荐类型，用户点击"添加到路线"后插入到当前节点之后

### 3.2 F-16 弹性路线规划

| 属性 | 描述 |
|------|------|
| **功能描述** | 允许用户在路线中标记"必去"和"可选"地点，一键生成不同紧凑度的路线方案 |
| **用户价值** | 现实中的出行计划常常变化，弹性规划让"临时不想去"不会毁掉整个路线 |
| **优先级** | T1 |
| **详细规则** | |

#### 3.2.1 必去 / 可选标记

- 路线中的每个地点可标记：
  - ⭐ **必去**（默认）：核心目的地，不可移除
  - ○ **可选**：感兴趣但不强求，时间充裕再去
- 标记方式：在路线排序列表中，点击地点旁的星标切换

#### 3.2.2 一键生成多方案

- **紧凑版**：包含所有必去 + 可选地点，按最近邻排序
- **轻松版**：仅包含必去地点，移除所有可选，适当加大停留间隔
- **用户选择**：生成后对比两个方案的总耗时和地点数，用户选择保存哪个

```
┌───────────────────────────────────────────────┐
│  生成路线方案                                  │
│  ─────────────────────────────────────────    │
│  📋 紧凑版（6个地点）                          │
│     游玩 5h · 交通 1h · 约 ¥180               │
│     [ 预览 ]  [ 保存为 Day1 ]                  │
│                                               │
│  📋 轻松版（4个地点）                          │
│     游玩 3h · 交通 40min · 约 ¥120            │
│     [ 预览 ]  [ 保存为 Day1 精简版 ]            │
└───────────────────────────────────────────────┘
```

#### 3.2.3 手动调整弹性

- 用户拖拽调整顺序时，可选地点与必去地点视觉区分
- 可选地点在路线上以 **虚线边框 + 浅色背景** 标识
- 支持一键"暂时隐藏可选地点"查看精简路线

### 3.3 F-17 多人轻协作

| 属性 | 描述 |
|------|------|
| **功能描述** | 生成一个可分享的路线链接，好友可查看、投票和提议新增地点 |
| **用户价值** | 和朋友一起规划出行是高频场景，但不想做复杂的账号体系 |
| **优先级** | T1 |
| **详细规则** | |

#### 3.3.1 分享机制

- 用户保存路线后，点击「分享」按钮
- 系统生成一个 **短链接 + 只读查看页**
- 查看页展示路线详情（地点列表、地图、时间预算）
- URL 格式：`https://yourapp.com/route/[short-id]`

#### 3.3.2 协作者能力（轻量）

- 查看页底部有「💬 留言建议」区域
- 协作者可以：
  - 👍 对已有地点投票"想去"
  - ➕ 提议新增一个地点（填写名称和地址）
- 提议的新地点进入「待确认」状态，路线创建者可在应用内查看并决定是否采纳

#### 3.3.3 技术实现

- **新增数据表**：

```sql
CREATE TABLE `route_collaborators` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `route_plan_id` BIGINT UNSIGNED NOT NULL,
  `nickname` VARCHAR(50) NOT NULL COMMENT '协作者昵称',
  `vote_data` JSON DEFAULT NULL COMMENT '投票数据',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  INDEX `idx_route` (`route_plan_id`),
  CONSTRAINT `fk_rc_route` FOREIGN KEY (`route_plan_id`) REFERENCES `route_plans` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='路线协作者表';

CREATE TABLE `route_suggestions` (
  `id` BIGINT UNSIGNED NOT NULL AUTO_INCREMENT,
  `route_plan_id` BIGINT UNSIGNED NOT NULL,
  `suggested_by` VARCHAR(50) NOT NULL COMMENT '提议者昵称',
  `place_name` VARCHAR(200) NOT NULL,
  `place_address` VARCHAR(500) DEFAULT NULL,
  `status` VARCHAR(20) NOT NULL DEFAULT 'pending' COMMENT 'pending/accepted/rejected',
  `created_at` DATETIME NOT NULL DEFAULT CURRENT_TIMESTAMP,
  PRIMARY KEY (`id`),
  INDEX `idx_route` (`route_plan_id`),
  CONSTRAINT `fk_rs_route` FOREIGN KEY (`route_plan_id`) REFERENCES `route_plans` (`id`) ON DELETE CASCADE
) ENGINE=InnoDB DEFAULT CHARSET=utf8mb4 COMMENT='路线地点提议表';
```

- 无需登录：使用 `nickname + 浏览器指纹` 作为临时身份标识
- 后端生成短 ID（如 `a3f7x2`），映射到 `route_plan_id`

## 4. T2 层：长期愿景（v2.0+ 方向）

### 4.1 F-18 城市记忆库

将工具从"规划未来"扩展为"记录过去"：

- **到访日记**：标记"已去过"的地点可补充到访日期、照片、短记
- **城市地图回顾**：自动生成"我的深圳地图 2026"，展示一年内去过的所有地方
- **年度总结**：类似 Spotify Wrapped，生成"你今年探索了 X 个新地方，最爱去的区域是 Y"

### 4.2 F-19 周期性发现

- **每周推荐**："这周你还没去过新地方，附近有 5 个你收藏但没去过的地点"
- **距离提醒**："你收藏的 XX 咖啡馆已经 3 个月没去了，要不要安排一下？"
- **探索挑战**："本月目标：去 4 个不同街道的新地方"

### 4.3 F-20 外部工具联动

让规划真正落地到出行：

| 联动目标 | 实现方式 | 用户价值 |
